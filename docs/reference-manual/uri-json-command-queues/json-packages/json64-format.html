JSON64 Format

JSON64 is a transport encoding for JSON Packages — the same JSON data, encoded as a Base64 string. This is useful in environments where raw JSON is difficult to pass cleanly, such as URL parameters that would require heavy percent-encoding, or inter-app communication channels that mangle special characters.

  • Encoding Rules

Why Base64?​

JSON contains characters — braces, quotes, colons, commas — that conflict with URL encoding, shell escaping, and other transport layers. Base64 encodes arbitrary data as a safe string of alphanumeric characters, plus +, /, and =. This means a JSON Package can be passed through a URI parameter or an automation tool without worrying about escaping.

The trade-off is readability — Base64 strings are opaque. JSON64 is not intended for hand-authored commands. It's designed for machine-to-machine communication, where an automation tool (like Apple Shortcuts or a shell script) generates the JSON, encodes it, and passes it to Obsidian.

Two JSON64 Parameters​

JSON64 encoding applies to two directive keys in the JSON Package structure:

ParameterDecoded EquivalentContains
jsonData64jsonDataA full JSON Package (directives + field data) encoded as Base64. Used with the fromJson command.
fieldData64fieldDataA field data object encoded as Base64. Used with the fieldData parameter.

Both work the same way: the plugin decodes the Base64 string, parses the resulting JSON, and proceeds as if the decoded equivalent had been passed directly.

Encoding Rules​

  • Encode the JSON string using standard Base64 (RFC 4648)
  • The JSON must be a valid object — not an array, primitive, or null
  • The decoded result follows the same validation rules as its non-encoded counterpart

Encoding Example​

Given this field data:

{"attendees": "Alice, Bob", "meetingType": "standup", "duration": 30}

The Base64 encoding is:

eyJhdHRlbmRlZXMiOiAiQWxpY2UsIEJvYiIsICJtZWV0aW5nVHlwZSI6ICJzdGFuZHVwIiwgImR1cmF0aW9uIjogMzB9

This can be passed as fieldData64 in a URI or JSON file.

Examples​

In a URI​

Instead of encoding a complex JSON object as a URL parameter:

obsidian://z2k-templates?cmd=new&templatePath=Templates/Meeting.md&fieldData64=eyJhdHRlbmRlZXMiOiAiQWxpY2UsIEJvYiIsICJtZWV0aW5nVHlwZSI6ICJzdGFuZHVwIn0=

This avoids the percent-encoding nightmare of passing raw JSON in a URL.

In a JSON Command File​

{
  "cmd": "fromJson",
  "jsonData64": "eyJjbWQiOiAibmV3IiwgInRlbXBsYXRlUGF0aCI6ICJUZW1wbGF0ZXMvTm90ZS5tZCIsICJ0aXRsZSI6ICJIZWxsbyJ9"
}

The jsonData64 value decodes to {"cmd": "new", "templatePath": "Templates/Note.md", "title": "Hello"}, which is then processed as a normal command.

Generated by Apple Shortcuts​

A typical automation flow:

  1. Apple Shortcuts collects data from the user or another app
  2. The shortcut builds a JSON object with the field values
  3. It Base64-encodes the JSON string
  4. It opens a URI with fieldData64=<encoded-string>
  5. Obsidian receives the URI and Z2K Templates decodes and processes it

For a complete walkthrough, see How-to Pass Create Files from Templates in Apple Shortcuts.

INTERNAL NOTES
  • The plugin accepts both standard Base64 (RFC 4648, using + and /) and URL-safe Base64 (using - and _). Either variant decodes correctly. URL-safe Base64 is preferable for URI transport since it avoids percent-encoding the + character.
