fieldData

The fieldData directive lets you bundle template field data separately from command directives in a JSON Package. Instead of mixing field names alongside cmd, templatePath, and other directives at the top level, you can group all field data into a single nested object — or point to an external file that contains it.

  • Merge Behavior
  • Validation
  • Examples

Three Input Forms​

The fieldData parameter accepts three forms of input, depending on context:

1. Inline JSON String​

A JSON string containing the field data object. Used primarily in URI Actions, where all parameter values are strings.

obsidian://z2k-templates?cmd=new&templatePath=Templates/Note.md&fieldData={"title":"My Note","rating":5}

The plugin detects a string that starts with { and parses it as JSON.

2. Vault-Relative File Path​

A string that does not start with { is treated as a vault-relative path to a .json file. The plugin reads and parses the file contents.

{
  "cmd": "new",
  "templatePath": "Templates/Contact Card.md",
  "fieldData": "Data/imports/new-contact.json"
}

The referenced file must contain a valid JSON object:

{
  "name": "Alice Chen",
  "role": "Engineer",
  "team": "Platform"
}

This is useful when field data is generated by an external tool and saved to the vault, or when the same data set needs to be reused across multiple commands.

3. Parsed Object​

When the JSON Package itself arrives from a JSON source (a .json or .jsonl file processed by the Command Queue), fieldData can be a native JSON object — already parsed, no string processing needed.

{
  "cmd": "new",
  "templatePath": "Templates/Note.md",
  "prompt": "none",
  "fieldData": {
    "title": "Inline Object",
    "category": "demo",
    "priority": 3
  }
}

This is the cleanest form — directives and field data are visually separated within the same file.

Merge Behavior​

When both top-level field data keys and fieldData are present, they are merged. Top-level keys take precedence.

{
  "cmd": "new",
  "templatePath": "Templates/Note.md",
  "title": "Override Title",
  "fieldData": {
    "title": "Original Title",
    "category": "demo"
  }
}

Result:

  • title = "Override Title" (top-level wins)
  • category = "demo" (from fieldData, no conflict)

This precedence rule means you can use fieldData as a base set of defaults and override individual fields at the top level.

Validation​

The plugin enforces these rules on the resolved fieldData value:

  • Must be a plain object — arrays and null are rejected
  • If provided as a string, it must be either valid JSON starting with { or a valid vault-relative file path
  • If provided as a file path, the file must exist in the vault and contain valid JSON
  • All values within the object are treated as field data — no directive keys are extracted from inside fieldData
note

Keys inside fieldData are always treated as field data. Placing a directive key like "prompt": "none" inside fieldData will not set the prompt mode — it will create a template field named prompt with the value "none". Directives must always appear at the top level of the JSON Package.

Examples​

Separating Concerns in a Command File​

{
  "cmd": "new",
  "templatePath": "Templates/Project Brief.md",
  "prompt": "remaining",
  "finalize": false,
  "fieldData": {
    "projectName": "Z2K v2",
    "owner": "GP",
    "startDate": "2024-01-15",
    "budget": 50000
  }
}

Directives (cmd, templatePath, prompt, finalize) are clearly separated from field data — no ambiguity about which keys are which.

Loading Field Data from a File​

{
  "cmd": "new",
  "templatePath": "Templates/Import Record.md",
  "prompt": "none",
  "finalize": true,
  "fieldData": "Data/imports/record-2024-01-15.json"
}

Overriding Specific Fields​

{
  "cmd": "new",
  "templatePath": "Templates/Contact.md",
  "name": "Updated Name",
  "fieldData": "Data/contacts/alice.json"
}

If alice.json contains {"name": "Alice Chen", "role": "Engineer"}, the result will be name = "Updated Name" and role = "Engineer".

INTERNAL NOTES
  • The file path detection heuristic is simple: if the string starts with {, it's inline JSON; otherwise it's a file path. This means a file path that starts with { (unusual but theoretically possible) would be misinterpreted as JSON. Worth noting as a known edge case.
