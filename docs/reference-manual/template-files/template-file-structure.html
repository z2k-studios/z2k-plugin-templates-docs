Template File Structure

Let's talk about the structure of a Template file. this needs serious editing

Markdown Format​

Because the only strict requirement for a template file is that it must be a text file, preferably markdown based, the file structure consists of a text file stored in standard straight text.

Like most Markdown formats, the template can optionally support YAML frontmatter to provide additional data and configuration for the file.

These follow the same rules as any Obsidian note. If you are unfamiliar with Obsidian frontmatter or Markdown structure, refer to Obsidian’s official documentation:

  • Obsidian Frontmatter: https://help.obsidian.md/Editing+and+formatting/Metadata
  • Obsidian Markdown: https://help.obsidian.md/Editing+and+formatting/Basic+formatting+syntax

This page focuses on how Z2K Templates interprets these structures.

YAML Frontmatter​

YAML frontmatter appears at the top of the template file and is enclosed between triple-dashes:

---
z2k_template_type: note
z2k_template_name: Sample Template
custom_field: "{{myField}}"
---

Frontmatter may contain:

  • Z2K configuration fields such as z2k_template_name or z2k_template_default_miss_handling.
  • Field definitions that include inline {{field-info}} calls.
  • Static metadata that should be copied directly to the final file.

The Z2K Template Engine parses frontmatter before body content. The presence of {{field-info}} entries inside YAML is permitted but constrained by YAML syntax rules; these must be valid YAML values.

Partials cannot contain frontmatter. Only top-level template files and system blocks may include YAML sections.

Built-in and Custom Fields in YAML​

Z2K Templates discovers field references inside YAML using its AST parser. This means that any variable written as {{someField}} in frontmatter will create a field definition unless overridden or configured via a {{field-info}} entry.

When both body and frontmatter use the same field name, Z2K merges the definitions following the precedence rules described in Built-In Fields and Prompting.

Relation to the Rendered File Title​

Z2K normalizes certain field names such as {{noteTitle}} and {{cardTitle}} into a canonical field {{fileTitle}}. This allows templates written before the alias change to continue functioning. The rendered file's title may be derived from:

  • The filename generated by the command triggering the template.
  • A {{fileTitle}} field resolved in YAML or body.
  • A z2k_template_default_title entry in YAML.

Templates should explicitly specify title logic when necessary.

Body Content​

The body contains Markdown mixed with template expressions. Everything written in the body is copied to the final file after fields have been resolved.

Example:

# Meeting Notes for {{date}}

Attendees: {{attendees}}

{{#if agenda}}
## Agenda
{{agenda}}
{{/if}}

The body allows:

  • Standard Markdown
  • Field references: {{fieldName}}
  • Helper functions: {{format-date date}}
  • Conditional and iterative Handlebars constructs

See Template Fields and Helper Functions for reference.

System Blocks and Global Blocks​

System Blocks and Global Blocks are mechanisms that prepend or inject content into your template before rendering. They are resolved before your main template content and can alter the YAML or body structure.

For example, a system block might inject standardized metadata:

---
author: "{{userName}}"
---

Or prepend structured Markdown:

> This note was created using the Standard Meeting Template

These behaviors are described in detail under System Block Templates.

Minimum Structure​

A template file must contain at least one of the following:

  • YAML frontmatter with metadata or fields
  • Body content with Markdown or fields

A template with no YAML and no fields is valid but not useful.

Examples:

Minimal Template​

---
---
Hello world

Practical Minimal Template​

---
z2k_template_type: note
---
# {{fileTitle}}
Created on {{date}}

Template with Fields in YAML​

---
project: "{{projectName}}"
owner: "{{owner}}"
---
# Project Kickoff
Welcome to {{projectName}}

Field Discovery and Parsing​

Z2K parses the template in several phases (see Lifecycle of a Template for full details):

  1. Extract YAML frontmatter.
  2. Extract system block frontmatter.
  3. Parse body.
  4. Determine all fields referenced.
  5. Merge field definitions from built-ins, system blocks, YAML, and body.
  6. Resolve missing fields via sequential prompting.

This page focuses on the structural aspect—how the template file is divided—not the full lifecycle.

Recommended Practices​

  • Keep YAML simple; offload logic to body fields or helper functions.
  • Place default values and directives near their associated fields using {{field-info}}.
  • Avoid overusing fields inside YAML unless necessary for metadata.
  • Prefer explicit titles using {{fileTitle}}.

Summary​

A template file is just Markdown with optional YAML, enriched with field syntax. Understanding this structure helps you reason about how templates are parsed, merged, and ultimately rendered by the Z2K Template Engine.

danger
  • This page assumes familiarity with YAML syntax; invalid YAML will cause template parsing errors.
  • Partial templates cannot have frontmatter; only main templates and system/global blocks can define YAML.
  • System block prepending may affect line numbers during parsing; see System Block Templates for deeper details.
